# Rescuer HQ — Realtime & WebSocket Feature Architecture
**Статус:** Production-Ready Specification  
**Модуль:** `core/network` & `features/realtime`

---

## 1. Realtime Core Architecture & Layers

Регистрируя и координируя спасательные операции, мы сочетаем однонаправленные потоки данных Clean Architecture с реактивной событийной моделью. Потокобезопасность сетевых интерфейсов обеспечивается жестким разделением ответственности.

```text
 ┌────────────────────────────────────────────────────────────────────────┐
 │                           PRESENTATION LAYER                           │
 │  - RealtimePresenceWidget (Визуальная сетка онлайн-статусов отрядов)    │
 │  - MapOverlayStreamer (Реактивная отрисовка треков волонтеров на карте) │
 │  - ChatStreamView (Интерактивная лента сообщений)                       │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                              DOMAIN LAYER                              │
 │  - RealtimeEvent (Строго типизованное иммутабельное событие)           │
 │  - ConnectionStatus (Enum: Connected, Disconnected, Reconnecting)      │
 │  - PresenceModel (Информационные слепкие активности и координат)       │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                               DATA LAYER                               │
 │  - WebSocketService (Центральный менеджер соединений и heartbeat-цикла)│
 │  - WebSocketEventDispatcher (Дедупликация событий, маршрутизация)       │
 │  - RealtimeRepository (Интерфейс подписок на комнаты, каналы и SOS)    │
 └────────────────────────────────────────────────────────────────────────┘
```

## 2. WebSocket Connection Lifecycle

Жизненный цикл сокета контролируется реактивно через авторизационный провайдер `authProvider`.

### Диаграмма переходов жизненного цикла (Symmetric Socket State Transitions)

```text
       User Authenticated
              │
              ▼
      ┌───────────────┐
      │  Connecting   │ ───► Authentication Ticket Handshake
      └───────┬───────┘
              ▼
      ┌───────────────┐
      │   Connected   │ ◄───┐ Heartbeat Loop
      └───────┬───────┘     │ (Ping: 30s ↔ Pong: 10s timeout)
              │             │
   Network Lost / Err / Dead Socket
              │             │
              ▼             │ Retries < Limit (Expo Backoff)
      ┌───────────────┐     │
      │ Reconnecting  │ ────┘
      └───────┬───────┘
              │
         Retries >= Limit / Forced Logout
              │
              ▼
      ┌───────────────┐
      │ Disconnected  │ ◄─── Stream Cleanup & Hard Memory Reset
      └───────────────┘
```

### Доверительная интеграция токенов (Token-Auth Integration)
- При инициализации сокета клиент запрашивает свежий `AccessToken` из `secureStorageProvider`.
- Токен передается первым кадром (Auth Ticket Validation) или через параметры соединения:
  `ws://api.rescuerhq.org/ws?token=<access_token>`.
- Если сокет ловит ошибку авторизации `401 Unauthorized`/`Expired Signature`, он временно закрывается с кодом `4001` (Auth Token Expired), запрашивает перегенерацию токена через интерцептор Dio и выполняет повторное подключение.

## 3. Realtime Event System & Formatting (Event Dispatcher)

Единый брокер событий (`WebSocketEventDispatcher`) обеспечивает синтаксический анализ полезной нагрузки, маршрутизацию по модулям (Chat, Maps, Tasks) и дедупликацию.

### Формат Сообщений (JSON Scheme Contract)
Каждое сообщение, передаваемое по сокету, строго удовлетворяет следующему JSON-контракту:
```json
{
  "uuid": "f81d4fae-7dec-11d0-a765-00a0c91e6bf6",
  "event_type": "chat.message.created",
  "timestamp": "2026-05-28T12:35:58Z",
  "room_id": "search_room_24-0912",
  "payload": {
    "sender_id": "913f99e4-fa86",
    "call_sign": "Байкал-50",
    "message_text": "Выхожу в квадрат А-2",
    "speech_to_text": "Вырою в квадрат о два"
  }
}
```

### Защита от дубликатов (Deduplication Layer)
Проблемы нестабильного радиосигнала и повторные отправки ведут к дублированию кадров. Диспетчер использует LRU-кэш на базе массива `_seenEventIds` емкостью 100 уникальных `UUID`. Если событие с входящим `uuid` уже присутствует в кэше, оно игнорируется, предотвращая лишние рендеры UI и дублирование чат-сообщений.

## 4. Reconnect Strategy (Exponential Backoff & Network Recovery)

При потере радиосвязи в лесных и удаленных районах включается автоматическое ступенчатое экспоненциальное восстановление:

$$\text{Delay} = \min(t_{\text{base}} \times 2^{\text{attempt}}, t_{\text{max}})$$

Где $t_{\text{base}} = 2\text{ сек}$, $t_{\text{max}} = 60\text{ сек}$.

### Автоматическое переподключение при восстановлении сети
Служба непрерывно следит за нативным состоянием сотовой сети (например, через плагин `connectivity_plus`). Переход из состояния `None` -> `Wireless/Cellular` служит безусловным триггером мгновенного обхода таймера ожидания и форсированного реконнекта.

## 5. Heartbeat & Dead Socket Recovery Policy

Для детекции «полумертвых» TCP-соединений (когда индикатор сети "горит", но интернет-канал заблокирован или зашумлен) реализован принудительный пинг-понг:

1. **Ping**: Каждые **30 секунд** клиент отправляет на бэкенд текстовый фрейм `{"type": "ping"}`.
2. **Pong Timeout**: Бэкенд обязан вернуть фрейм `{"type": "pong"}` в течение **10 секунд**.
3. **Dead Socket Cleanup**: Если ответ не поступил вовремя, сокет принудительно разрывается методом `.close()`, вызывая штатную цепочку переподключения `Reconnecting`.

## 6. App Lifecycle Handling & Resource Optimization

Учет фоновых режимов для оптимизации батареи в полевых операциях:

* **Foreground**: Полная оперативная синхронизация. Все каналы подписки (чаты, интерактивные карты, геотрекинг коллег) активны. Частота пинга 30 секунд.
* **Background**: При уходе приложения в фон (считывается через `WidgetsBindingObserver`) сокет отключает некритические комнаты (например, каналы присутствия и детальные карты квадратов), чтобы сберечь заряд аккумулятора.
* **SOS Exception**: Если на устройстве активирован режим спасения **SOS Emergency** (высокая частота GPS-пинга), то оффлайновые ограничения на сокет игнорируются! Сокет удерживает беспрерывный приоритетный канал отправки координат даже в фоновом режиме.

## 7. Presence System & Multi-device Tracker

Система контролирует интерактивное присутствие отрядов на поле:
* **Online Registry**: При успешном коннекте отправляется событие `volunteer.presence.check-in` с радио-позывным волонтера.
* **Last-Seen Monitor**: Бэкенд транслирует статус присутствия `user_presence.updated`.
* **Multi-device Protection**: Идентификация сессии связывается с UUID конкретного устройства, исключая наложение треков при одновременной авторизации на планшете координатора и личном андроид-смартфоне волонтера.
