## 2. State Management Architecture (Архитектура управления состоянием на Riverpod)

Управление состоянием в приложении реализуется строго декларативно с использованием реактивных провайдеров **Riverpod 2.x**. Потоки данных строго однонаправленны. Состояния иммутабельны и генерируются библиотекой **Freezed**.

```text
                      ┌──────────────────────┐
                      │    User Trigger      │
                      │ (Нажатие кнопки SOS) │
                      └──────────┬───────────┘
                                 ▼
                     ┌────────────────────────┐
                     │   SosNotifier (State)  │ ◄─── Наблюдает за LocationNotifier
                     └──────────┬─────────────┘
                                 ▼
                     ┌────────────────────────┐
                     │ WebSocketService (Core)│ ◄─── Сериализует событие sos.created
                     └──────────┬─────────────┘
                                 ▼
                     ┌────────────────────────┐
                     │   Drift SQLite Cache   │ ◄─── Сохраняет копию локальной метки
                     └────────────────────────┘
```

### Основные слои провайдеров
1. **Repository Providers (Слой данных):**
   * `databaseProvider`: Синглтон Drift-клиента `AppDatabase`.
   * `secureStorageProvider`: Объект безопасного шифрованного хранилища `FlutterSecureStorage`.
   * `authRepositoryProvider`: Реализует отправку авторизационных пакетов и хранение JWT-сессии.
2. **WebSocket & Network Providers (Слой транспорта):**
   * `dioClientProvider`: Настроенный инстанс Dio с набором интерцепторов.
   * `webSocketServiceProvider`: Управляет жизненным циклом сокет-канала, генерирует стрим событий для других систем.
3. **Core State Flow Notifiers (Бизнес-логика):**
   * `authProvider (StateNotifierProvider<AuthNotifier, AuthState>)`:
     * Наблюдает за состоянием жизненного цикла пользователя. Инициализирует редирект на экран авторизации при отзыве JWT.
   * `locationNotifierProvider (StateNotifierProvider<LocationNotifier, LocationState>)`:
     * Запускает фоновый стрим геолокации. Реализует буферизацию пакетов координат в Drift SQLite и отправляет их в реальном времени на сокет `user.location.updated`.
   * `syncServiceProvider (Provider<SyncService>)`:
     * Оркестратор отложенного фонового кэша. Прослушивает изменение сетевого статуса `connectivityProvider` и инициирует фоновый сброс накопленных событий (аутбокс чата, треков геолокации) на бэкенд.

## 3. API Layer Architecture (Архитектура сетевого взаимодействия Dio)

Сетевое взаимодействие осуществляется через защищенный обернутый экземпляр **Dio**. Ошибки обрабатываются на самом раннем этапе с конвертацией в типизированные исключения `AppException`.

### Паттерн Token Rotation (Многопоточный перехват и обновление JWT):
В случае получения `HTTP 401 Unauthorized` интерцептор выполняет атомарный перезапуск сессии:

1. **Блокировка очереди (Locking):** Сетевой поток Dio останавливает отправку всех последующих исходящих запросов.
2. **Вызов Refresh API:** Выполняется POST запрос на `/api/v1/auth/refresh` с использованием ранее сохраненного шифрованного `refresh_token`.
3. **Обновление хранилища:** Новые accessToken и refreshToken записываются в Keychain/SharedPrefs через безопасное хранилище.
4. **Коммит и Повтор (Retry):**
   * Прерванный запрос модифицируется новым заголовком `'Authorization': 'Bearer <new_token>'`.
   * Dio повторно выполняет оригинальный запрос.
   * Очередь Dio разблокируется для продолжения штатного обмена данными.
5. **Force Logout:** Если операция обновления токена завершилась ошибкой (токен устарел или отозван администратором), вызывается очистка локального кэша и пользователь безопасно перенаправляется на экран логина.

## 4. WebSocket Architecture (Архитектура реального времени)

Реалтайм-шина разработана на базе пакета `web_socket_channel`. Работа сокетов защищена от разрывов связи в лесистой местности.

* **Авторизационное рукопожатие (Handshake):**
  Токен передается при инициализации соединения по протоколу WS (`wss://api.rescuerhq.org/ws?token=...`). Бэкенд валидирует сессию перед апгрейдом протокола.
* **Heartbeat (Пинг-понг интервал):**
  Каждые 30 секунд со стороны клиента шлется пакет `{"event": "ping"}`. Приложение ожидает получение ответа `{"event": "pong"}` в течение 10 секунд. При отсутствии ответа соединение принудительно разрывается текущим сеансом для запуска реконнекта.
* **Стратегия отложенного реконнекта (Exponential Backoff):**
  При отключении запуск повторного подключения рассчитывается по формуле:
  $$\text{Delay} = \min(2^{\text{Attempt}} \times 2, 60) \text{ секунд}$$
  Данный алгоритм снижает избыточную нагрузку на сотовые вышки в зонах со слабым сигналом.
* **Stream-Based Маршрутизация:**
  Служба `WebSocketService` предоставляет широковещательный поток событий спасательной операции:
  ```dart
  webSocketService.eventStream
    .where((event) => event['event'] == 'chat.message.created')
    .listen(_handleIncomingMessage);
  ```

## 5. Offline-First Architecture & Sync Engine

Приложение спасателя функционирует автономно без подключения к Интернету. База данных Drift выступает единым источником правды (**Single Source of Truth**):

1. **Оптимистичные обновления (Optimistic Updates):**
   При отправке сообщения или изменении статуса задачи запись мгновенно создается на локальном SQLite со флагом `sync_status = pending` и отображается в UI. Пользователь видит мгновенный отклик приложения.
2. **Транзакционная очередь (Persistent Outbox):**
   Все мутации состояния системы заносятся во внутреннюю таблицу миграций `SyncQueueTable` с указанием порядкового номера, типа операции и полезной нагрузки JSON.
3. **Разрешение конфликтов (Conflict Resolution):**
   * **Locations:** Данные всегда добавляются кумулятивно как хронологический лог (логический конфликт невозможен).
   * **Districts / Задачи спасения:** Бэкендированное значение имеет приоритет. В случае изменения границ координируемых секторов, локальный клиент накатывает серверную информацию сверху по таймстампу `updated_at` (Last-Write-Wins).

## 6. Local Database Architecture ( Drift SQLite)

Мы используем развитый реляционный ORM **Drift (Moor)** поверх нативных библиотек SQLite.

### Таблицы базы данных приложения:
* **`UsersTable`:** Идентификатор спасателя, ФИО, телефон, роль доступа (Director, Coordinator, Volunteer) и статус одобрения IT-администратором.
* **`SearchesTable`:** Карточки поисковых операций (id, имя пропавшего ребенка, статус подготовки, дата создания, активные гео-метки центра поиска).
* **`LocationQueueTable`:** Буфер для фоновой записи треков геолокации. Сохраняет широту, долготу, погрешность GPS, скорость и время записи. Наличие индекса по колонке `synced` ускоряет выборку несохраненных данных на сотовую вышку.
* **`ChatQueueTable`:** Временная локальная очередь отправки сообщений в групповые чаты поисковиков.

```sql
-- Пример внутренней схемы миграции и структуры Drift
CREATE TABLE location_queue (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    latitude REAL NOT NULL,
    longitude REAL NOT NULL,
    accuracy REAL NOT NULL,
    timestamp INTEGER NOT NULL,
    synced INTEGER DEFAULT 0
);
CREATE INDEX idx_location_queue_synced ON location_queue (synced);
```

## 7. Authentication Flow (Безопасность и сессии событий)

Поскольку координация поиска связана с чувствительной информацией, хранение учетных данных защищено аппаратно:

* **Шифрование токенов (`flutter_secure_storage`):**
  * **iOS:** Ключи пишутся напрямую в безопасный системный защищенный сервис Keychain с флагом `kSecAttrAccessibleAfterFirstUnlock`.
  * **Android:** Данные сохраняются через механизм защищенных настроек `EncryptedSharedPreferences`, где ключи шифруются по стандарту AES-256 в аппаратном KeyStore устройства.
* **Очистка данных при выходе (Logout Cleanup):**
  При нажатии на кнопку "Выйти" или удаленном отзыве прав:
  1. Вызовы фонового GPS-трекинга немедленно прекращаются.
  2. WebSocket-соединение закрывается с флагом `clean_close`.
  3. Токены удаляются из Keychain спасателя.
  4. Мутирующие таблицы локальной базы данных Drift форматируются (Cascade Truncate).

## 8. Maps & GIS Architecture (ГИС Архитектура)

Картографический оверлей спасательного приложения — самый высоконагруженный визуальный элемент.

* **Геометрия зон контроля (District Polygons):**
  Векторные границы секторов поиска (построены координатором на штабном пульте) парсятся из GeoJSON-массивов и рисуются на поверхности карты в виде оверлеев `Polygon` с настраиваемой прозрачностью и цветовым кодом сложности сектора.
* **Кластеризация маркеров (Marker Clustering):**
  Для компенсации падения FPS при выводе сотен одновременно ищущих волонтеров на экране карты применяется алгоритм кластеризации. Спасатели, находящиеся в одной области, объединяются в единый интерактивный маркер-круг с цифрой-счетчиком численности.
* **SOS Пин-маркеры:**
  Генерируют яркие красные пульсирующие маркеры с вызовом обратной связи по нажатию. При переключении в SOS-режим на карте включается динамический рендеринг затухающих окружностей (эффект волны радарного пинга).

## 9. Media Architecture (Сжатие и загрузка медиа по воздуху)

Экстренная отправка фото и аудио отчетов в полевых условиях:

* **Адаптивное аудиосжатие:**
  Устройство записывает речевые заметки спасателя в формате `.m4a` или `.ogg` (битрейт 16-22 kbps, моно-канал). Это минимизирует итоговые размеры файлов до 15-30 Кбайт, позволяя загрузить файлы даже через базовые каналы 2G/EDGE.
* **Фоновый загрузчик отчетов (Background Uploader):**
  Изображения с места обнаружения следов пакуются и пережимаются до 1080p, после чего отправляются через отложенные фоновые планировщики, восстанавливающие передачу чанка при непредвиденном обрыве связи.

## 10. Push Notifications (FCM / Уведомления)

* **Режим тревоги Critical SOS:** Приходит специальный тип Push-уведомления от Firebase Cloud Messaging с высоким приоритетом (`priority: high` / `mutable_content: true`). В мобильном коде запускается нативный проигрыватель тревожной сирены, минуя беззвучные пресеты Android/iOS (через Critical Notifications API).
* **События фонового обновления метаданных:** Беззвучные (silent) пуши дают команду локальной СУБД обновить карту местности скрытно для волонтера, чтобы избежать траты времени по прибытию в оперативную зону.

## 11. Navigation & RBAC (Маршрутизация GoRouter)

Вся навигационная структура описана декларативным образом через `GoRouter`. Реализована строгая проверка прав на этапе перехода по адресам:

* **Маршруты, защищенные ролями спасения:**
  Экраны распределения районов поиска и управления волонтерскими группами закрыты охранной схемой. Дайверидж сессий сверяет JWT-токен пользователя. Если в claim-параметрах волонтера стоит статус `UserRole.volunteer`, GoRouter блокирует доступ к штабному интерфейсу оператора с мгновенным возвратом на главный экран.

## 12. Production Readiness (Продуктовая надежность)

* **Глобальный барьер сбоев:** Приложение упаковывается в отказоустойчивый контекст `runZonedGuarded`. Все ошибки времени выполнения отлавливаются без отображения серого экрана смерти устройства.
* **Динамический профиль батареи устройства:**
  Потребление энергии адаптируется по следующим условиям:
  * **Стандартный режим:** Точность высокая, фиксация координат каждые 30 секунд.
  * **Режим паники (SOS Active):** Повышение интервала GPS до 5 секунд для мгновенной пеленгации спасателя.
  * **Низкий уровень заряда (<15% батареи):** Снижение интенсивности опроса датчиков до 120 секунд с отключением пробуждения процессора (WakeLocks), отправка предупреждения о разряде в штаб HQ.

## 13. Сводная архитектурная схема Foundation

Ниже приведена иерархия слоев Flutter Foundation, которая реализована в структуре skeleton-кода проекта:

```text
 ┌────────────────────────────────────────────────────────┐
 │                      MaterialApp                       │
 ├────────────────────────────────────────────────────────┤
 │   Presentation Layer (UI Screens, Google Maps, Chat)   │
 ├────────────────────────────────────────────────────────┤
 │    Riverpod Controllers (LocationState, AuthState)     │
 ├────────────────────────────────────────────────────────┤
 │ Domain Layer (Repositories, Interfaces, Business logic)│
 ├────────────────────────────────────────────────────────┤
 │    Data Layer (Dio Client, Drift SQLite Database,      │
 │        WebSocket Manager, Secure Token Storage)        │
 └────────────────────────────────────────────────────────┘
```
