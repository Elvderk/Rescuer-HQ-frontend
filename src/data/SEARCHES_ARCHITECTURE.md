# Rescuer HQ — Searches & Rescue Resource Operations Architecture
**Статус:** Production-Ready Specification  
**Слой:** `features/searches`  
**Технологический стек:** Riverpod, Drift SQLite, Google Maps/Yandex MapKit Dart bindings, WebSocket, Dio API client

---

## 1. Feature Architecture Overview

Управление поисковыми операциями (объект `Search`) выражено в Clean Architecture с разделением на четыре классических слоя. Все изменения синхронизируются в реальном времени, а интерфейс работает по принципу автономности (Local-First).

```text
 ┌────────────────────────────────────────────────────────────────────────┐
 │                           PRESENTATION LAYER                           │
 │  - SearchesListScreen (Пагинированный грид с фильтрацией и оффлайн-кэшем)│
 │  - SearchDetailsScreen (Реактивная карточка пропавшего с картами и SOS)  │
 │  - CreateSearchScreen (Пошаговый мастер создания карточки пропавшего)    │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                           APPLICATION LAYER                            │
 │  - SearchesNotifier (Бизнес-логика, пагинация, фильтрация, мутации)     │
 │  - ActiveSearchNotifier (Синхронизация выбранного активного поиска)     │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                              DOMAIN LAYER                              │
 │  - SearchModel (Типизированная сущность пропавшего с приметными полями)│

 │  - SearchesCacheDataSource (Локальное зеркало поиска во встроенной БД) │
 └────────────────────────────────────────────────────────────────────────┘
```

## 2. State Management Architecture & Riverpod Providers

Сложная бизнес-логика поиска разбита на модульные реактивные граф-провайдеры:

* **`searchesFilterProvider`**: Хранит текущие параметры фильтрации (`SearchFilter` со свойствами `status`, `query`, `mineOnly`, `nearby`).
* **`searchesPaginationProvider`**: Осуществляет пагинацию, запрашивая списки по 20 записей на страницу.
* **`searchesNotifierProvider`**: Стейт-нотификатор (StateNotifier/Notifier), управляющий списком поисков. Синхронизирует данные между базой данных SQLite и удаленным эндпоинтом.
* **`activeSearchIdProvider`**: Удерживает ID выбранной в текущий момент поисковой операции. Используется сокетом для автоматического переподключения к комнатам чата и районов.
* **`activeSearchProvider`**: Соединяет `activeSearchIdProvider` и репозиторий для динамической подгрузки детальных параметров.

## 3. Realtime Events & WebSocket Handlers Integration

Когда координатор в штабеHQ меняет статус поиска или добавляет ориентировку, мобильный клиент мгновенно подхватывает изменения по WebSocket.

* **Регистрация обработчиков**:
  Мобильный клиент в методе `connect()` подписывается на события через `WebSocketEventDispatcher`:
  - `search.created` -> Добавляет новую карточку поиска в Drift DB и уведомляет UI.
  - `search.updated` / `search.status.changed` -> Обновляет локальную запись в Drift DB; если ID совпадает с активным поиском, обновляет рендер деталей.
  - `participant.joined` / `participant.left` -> Изменяет состав участников в локальном хранилище активной операции.

## 4. Offline first & Local Cache Sync Integration

Режим автономности гарантирует, что спасатель в глубоком лесу всегда имеет под рукой приметы пропавшего ребенка и контакты коллег:

1. **Local storage first**: Метод `loadSearches()` всегда сначала вычитывает кэш из Drift `SearchesTable`. Если интернет присутствует, репозиторий запрашивает бэкенд `/api/v1/searches`, сливает дельту (Merge) и инвалидирует устаревший кэш.
2. **Offline Drafts**: Конструктор `CreateSearchScreen` сохраняет черновик в локальную БД. Если во время публикации в лесу связь сорвалась, черновик встает в `SyncQueueTable` с присвоенным `idempotencyKey`. Как только телефон находит соту, фоновый движок заливает его на бэкенд.

## 5. Maps & Coordination Overlays Integration

Экран `SearchDetailsScreen` содержит встроенный интуитивный интерактивный виджет карты (Google Maps SDK / Yandex MapKit):

* **HQ Center**: Маркер в виде красного палаточного лагеря символизирует Координационный штаб сбора волонтеров. Координаты подтягиваются из карточки спасательного контракта.
* **Volunteer Locations Stream**: Реактивный слой поверх карт подписывается на стрим обновлений координат `locationUpdatesStreamProvider`. Волонтеры отображаются со своими позывными («Амур-12», «Заря-4»).
* **SOS Markers Overlay**: При поступлении события `sos.created`, на карте загорается пульсирующий ярко-красный SOS маркер с точными координатами бедствия и кнопкой построения маршрута эвакуации.
* **Districts Layout Overlay**: Границы назначенного спасательного района (Polygons) лениво рендерятся поверх подложки GeoJSON маски.

## 6. Navigation Flow & Role Security Handshaking

* **Searches Dashboard** (`/searches`) — главный экран перечня активных и закрытых операций.
* **Search Details** (`/searches/:id`) — детальная карточка.
* **Create Search Form** (`/searches/create`) — пошаговая форма создания. Доступ жестко преграждается на уровне GoRouter (`redirect` защита) и рендера UI: кнопка добавления видна только ролям `Coordinator`, `Senior Coordinator`, `Director`. При попытке взлома верстки запросы репозитория отвергаются бэкендом (403 Forbidden).
