# Rescuer HQ — Realtime Chat, Speech-to-Text & Media Synchronization Architecture
**Статус:** Production-Ready Specification  
**Слой:** `features/chat`  
**Технологический стек:** Drift (SQLite), WebSocket, Riverpod, MinIO/S3 API, Whisper/STT Engine API

## 1. Chat & Messaging Multi-Channel Architecture

В спасательном отряде важно структурировать потоки связи. Архитектура чата делит каналы на три критические группы:
- **Search Scope Channels**: Чаты, создаваемые автоматически под каждый активный поиск. Доступны всем волонтерам, прибывшим в квадрат операции.
- **Direct Private Rooms**: Каналы личного общения между координаторами или со следопытами для тонких тактических указаний.
- **Emergency / Broadcast Channels**: Административные каналы для мгновенного донесения общих приказов ("Всем вернуться в Штаб сбора").

```text
 ┌────────────────────────────────────────────────────────────────────────┐
 │                           PRESENTATION LAYER                           │
 │  - ChatsListScreen (Перечень доступных каналов с unread-счетчиками)    │
 │  - ChatScreen (Реактивная лента сообщений, кружков, аудио и реакций)   │
 │  - VoiceRecorderWidget (Запись аудиосообщений с динамической волной)    │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                           APPLICATION LAYER                            │
 │  - ChatRoomsNotifier (Вычисление unread-счетчиков и списка комнат)       │
 │  - ChatMessagesNotifier (Пагинированный бесконечной список сообщений)   │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                              DOMAIN LAYER                              │
 │  - ChatRoom / ChatMessage (Сущности компрессии медиа и типов)          │
 │  - MessageType (Text, Image, Voice, VideoNote, LiveLocation, System)   │
 │  - MessageStatus (Pending, Sent, Delivered, Read, Failed)             │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                               DATA LAYER                               │
 │  - ChatRepository (Слияние Drift SQLite и обмена сокетами)             │
 │  - MediaUploadQueue (Микро-очередь сжатия картинок и фоновой загрузки) │
 └────────────────────────────────────────────────────────────────────────┘
```

## 2. Dynamic WebSocket Frame Contract & Sequence Preservation

Для сохранения порядка репликации сообщений (Race Condition Prevention) и работы под нагрузкой:
- Каждое сообщение снабжается `client_uuid` (генерируется на телефоне) и серверным `sequence_id` (Auto-increment Long на стороне БД бэкенда).
- При получении пакетов клиент проверяет последовательность: если при переподключении обнаружен пропуск (например, локальный `sequence_id = 112`, а пришёл `sequence_id = 116`), клиент запрашивает пакетную дельту истории (Incremental Catch-up) по REST: `/api/v1/chats/:id/messages?after=112`.

## 3. Optimistic Media Processing & Background Outbox Pipeline

Медиафайлы (фотографии поисков, кружки, записи рации) обрабатываются в асинхронном конвейере:

```text
  [Record / Select Image]
             │
             ▼
  ┌────────────────────────┐
  │ 1. Local Compress Engine│ (Быстрое сжатие до 1200px / битрейт 96kbps)
  └──────────┬─────────────┘
             ▼
  ┌────────────────────────┐
  │ 2. Insert SQLite Row   │ ◄── Status: PENDING, local_uri: "/data/cache/audio.m4a"
  └──────────┬─────────────┘
             ▼
  ┌────────────────────────┐
  │ 3. Push to UploadQueue │ ◄── Retries with exponential backoff
  └──────────┬─────────────┘
             ▼
        Network Active?
       /               \
     Да                 Нет
     /                    \
    ▼                      ▼
┌───────────────────────┐ ┌──────────────────────────────────────────────┐
│  Upload directly to   │ │ Hold file in offline queue.                  │
│  MinIO/S3 bucket      │ │ Retry instantly upon state sensor trigger.   │
└──────────┬────────────┘ └──────────────────────────────────────────────┘
           ▼
┌───────────────────────┐
│ Swap local_uri -> URL │
│ Update status: SENT   │
└───────────────────────┘
```

## 4. Circular Video Messages & Voice Processing (Kruglyaky & STT)

- **Circular video notes («Кружки»)**: Записываются через контроллер камеры со строгим соотношением сторон 1:1, пережимаются в MP4 с низким битрейтом аудио и рендерятся в плеере через контейнеры с обрезкой границ клипом `ClipOval`.
- **Speech-to-Text (Интеграция с Whisper)**: Голосовые сообщения, отправляемые координаторам в поле, автоматически обрабатываются Whisper API. В чате под плеером отображается текстовая расшифровка (`VoiceTranscription`). Расшифровка сохраняется в Drift базе, что позволяет легко искать важные директивы в истории даже в оффлайн-режиме.

## 5. Rich Multi-User Features: Typing Indicators & Reactions

Для поддержания эффекта реального времени:
- **Typing Indicator**: Каждые 3 секунды при вводе текста волонтером сокет отправляет фрейм `chat.typing`. Другие участники видят анимированную плашку «Иванова вводит текст...». Счётчик сбрасывается по таймеру отсутствия обновлений (через 4 секунды).
- **Read Receipts**: При рендеринге свежих сообщений на экране, репозиторий отправляет пачку отметок о прочтении `chat.read` с указанием `max_sequence_id`.
- **Reactions & Replies Map**: Реакции хранятся в Drift в виде JSON мапы (например, `{"👍": ["user_1", "user_2"]}`).
