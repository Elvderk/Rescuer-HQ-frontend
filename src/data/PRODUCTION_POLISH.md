## 1. Full Production Audit Report

Аудит архитектуры Rescuer HQ выявил ключевые требования стабильности системы координации в сценариях с пропадающим покрытием сети (леса, овраги, горная местность):

| Компонент | Текущее Состояние | Целевое Production Состояние | Риски |
| :--- | :--- | :--- | :--- |
| **Репликация WebSocket** | Сырой стрим событий | Защищённый Heartbeat (Keep-Alive), автореконнект с экспоненциальным бэкоффом, дедупликация входящих UUID-фреймов. | Потеря критических сообщений, дублирование логов. |
| **Фоновое Геопозиционирование** | Стандартный логер координат | Высокоприоритетный Android Foreground Service с системным уведомлением. Изменение частоты в зависимости от SOS. | Убийство процесса планировщиком ОС (Doze Mode). |
| **Оффлайн Буфер (Drift)** | Линейное закладывание в очередь | Идемпотентная очередь `SyncQueueTable` с автоматическим слиянием конфликтов по схеме LWW (Last-Write-Wins). | Нарушение целостности реляционной базы. |
| **Рендеринг ГИС Карт** | Отрисовка сырых маркеров | Кластеризация маркеров поисковиков (Marker Clustering) и использование упрощённых прореженных полигонов секторов. | Утечки видеопамяти при >100 волонтерах на карте. |

## 2. Performance Optimization Plan (Реактивность & Rebuilds)

Для предотвращения лагов на устройствах эконом-класса полевых волонтеров внедряются следующие правила:
- **Riverpod Fine-Grained Filtering**: Вместо подписки на целый объект `TaskModel` виджеты подписываются только на необходимые свойства с помощью селекторов `ref.watch(tasksListProvider.select((list) => list.length))`. Это предотвращает перерисовку всей карточки при обновлении невидимых полей.
- **Virtualized ListView**: Использование `ListView.builder` с фиксированными размерами элементов (`itemExtent` или `prototypeItem`) для исключения рывков при бесконечной пагинации ленты сообщений чата.
- **Image/Media Pipeline Compression**: Предварительное сжатие фотографий со штаба с оригинального разрешения до 1280px по длинной стороне, а аудиодорожек рации — до битрейта 64-96 kbps (AAC-HE) перед отправкой в оффлайн-последовательность загрузки в S3.

## 4. WebSocket Stabilization Plan (Защищенность соединений)

Устойчивость сокетов обеспечивается на транспортном уровне:
1. **Heartbeat Protocol**: Каждые 20 секунд клиент посылает пустой фрейм `ping`. Если сервер не ответит фреймом `pong` в течение 10 секунд, соединение принудительно разрывается и объявляется автопереподключение.
2. **Exponential Backoff**: Попытки реконнекта следуют по схеме: $T_{next} = \min(2^n \times 1\text{c}, 60\text{с})$, предотвращая перегрузку сервера HQ (DDOS) при массовом выходе поисковиков из слепой зоны связи.
3. **Event Deduplication Cache**: Локальный FIFO-буфер хранит последние 100 обработанных UUID событий сокетов. Дубликаты, пришедшие из-за повторных сетевых рукопожатий, мгновенно отсекаются.

## 5. Offline Sync Finalization Plan (Разрешение конфликтов)

При выходе группы из леса и подключении к мобильной вышке оффлайн-очередь Drift обрабатывается по правилам:
- **Last-Write-Wins (LWW)**: Если координатор обновил задачу со штабного компьютера, а волонтер локально завершил её в оффлайне, при синхронизации сервер сравнивает время `updated_at`. Локальные завершенные отчеты объединяются с метаданными.
- **Idempotency Guarantee**: Все оффлайн вызовы мутаций снабжаются жестким UUID ключом идемпотентности, защищающим базу данных PostgreSQL на бэкенде от дублирования сообщений или повторных списаний задач.

## 6. Security Hardening (Безопасность полевой связи)

Защита данных поискового отряда строго контролируется:
- **Encrypted Token Store**: Токены авторизации JWT сохраняются исключительно в `flutter_secure_storage` (используя Keychain в iOS и Keystore во встроенном контейнере Android с аппаратным шифрованием AES-256).
- **TLS 1.3 pinning**: Принудительное шифрование трафика Dio со штабом, предотвращающее перехват геоданных волонтеров и приватных улик поисков посторонними лицами.

### Build Flavors & Permissions Justifications
Определяются 3 флейвора (`dev`, `staging`, `prod`) с разделением баз данных Firebase.  
В файле `AndroidManifest.xml` для публикации декларируются прозрачные обоснования использования геопозиции:
- `ACCESS_FINE_LOCATION`: Необходим для позиционирования волонтера на интерактивной карте поисков и ведения трека.
- `ACCESS_BACKGROUND_LOCATION`: Используется исключительно для отправки координат SOS при заблокированном телефоне поисковика.
- `RECORD_AUDIO`: Используется для записи голосовых сообщений и рации для распознавания Whisper Whisper.
