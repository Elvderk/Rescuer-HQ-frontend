# Rescuer HQ — Offline-First & Local Database Architecture
**Статус:** Production-Ready Specification  
**Слой:** `core/database` & `core/services/sync`  
**Технологический стек:** Drift (SQLite), ConnectivityPlus, FlutterSecureStorage

---

## 1. Offline-First Architecture & Data Access Pattern

В условиях поисково-спасательных работ в лесистой или горной местности сотовая связь нестабильна или отсутствует полностью. Приложение Rescuer HQ спроектировано по принципу **Local-First**:
- **Single Source of Truth (SSOT)**: Все экраны презентационного слоя (Чат, Список поисков, Спецификации задач, Границы районов) подписываются только на обновления локальной базы данных **Drift SQLite**.
- **Декларативные стримы (Reactive Streams)**: При изменении данных базовые DAO автоматически генерируют реактивные пакеты обновления в UI, избавляя от необходимости ручного дёргания Pull-To-Refresh.
- **Background Replication**: Сетевой уровень работает в изолированном фоновом режиме, скачивая дельты изменений и складывая их в локальную базу данных, а также вынимая из локального ящика (`SyncQueueTable`) накопленные оффлайн-мутации для отправки в штаб HQ.

## 2. Drift Database Schema & Table Relationships

Мы используем развитую реляционную структуру надстройки Drift. База данных содержит индексы для оптимизации географических запросов волонтеров и каскадное удаление данных для защиты конфиденциальности при разлогировании.

```text
 ┌──────────────────────┐           ┌──────────────────────┐
 │     UsersTable       │           │     SearchesTable    │
 ├──────────────────────┤           ├──────────────────────┤
 │ pk: id               │           │ pk: id               │
 │ role                 │           │ status (active/done) │
 └──────────┬───────────┘           └──────────┬───────────┘
            │                                  │
            │ 1                                │ 1
            ▼ *                                ▼ *
 ┌──────────────────────┐           ┌──────────────────────┐
 │     ChatsTable       │           │     TasksTable       │
 ├──────────────────────┤           ├──────────────────────┤
 │ pk: id               │           │ pk: id               │
 │ fk: search_id ───────┼──────────►│ fk: search_id        │
 └──────────┬───────────┘           └──────────────────────┘
            │
            │ 1
            ▼ *
 ┌──────────────────────┐           ┌──────────────────────┐
 │   ChatMessagesTable  │           │   LocationQueueTable │
 ├──────────────────────┤           ├──────────────────────┤
 │ pk: id               │           │ pk: id               │
 │ fk: chat_id          │           │ synced (0/1 index)   │
 │ sync_status (pending)│           └──────────────────────┘
 └──────────────────────┘
```

### Спецификации ключевых полей БД:
* **`LocationQueueTable`**: Поля `latitude`, `longitude`, `accuracy`, `timestamp`, `synced`. Создан составной B-Tree индекс `idx_location_synced` для извлечения неотправленных координат.
* **`SyncQueueTable`**: Очередь оффлайн-действий. Поля: `id` (autoincrement), `idempotency_key` (UUID), `action_type` (создание сообщения, смена статуса поиска, SOS-сигнал), `payload_json` (сериализованные параметры), `created_at`, `retry_count`.

## 3. Sync Engine & Processing Pipeline

Движок репликации (`SyncEngine`) работает по принципу конечного автомата (finite state machine):

```text
       Network Restored Notification
                    │
                    ▼
       ┌────────────────────────┐
       │   Acquire State Lock   │ (Предотвращает одновременные сеансы синхронизации)
       └────────────┬───────────┘
                    ▼
       ┌────────────────────────┐
       │ Read SyncQueue (LIMIT) │ ◄── Read ordered by created_at
       └────────────┬───────────┘
                    ▼
       ┌────────────────────────┐
       │  Execute HTTP / REST   │ (Отправка на бэкенд с Idempotency-Key)
       └────────────┬───────────┘
                    ├────────────────────────────────────┐
                    ▼ (Success - 200/201)                ▼ (Failed - Network/Server)
       ┌────────────────────────┐            ┌────────────────────────┐
       │   Delete Queue Row     │            │ Increment Retry Count  │
       └────────────┬───────────┘            └───────────┬────────────┘
                    │                                    ▼
                    │                     [Retry < Max?] ──► Expo-backoff delay
                    ▼                                    └──► Errored Outbox
         [Queue Empty?] ──► Release Lock
```

## 4. Optimistic Updates & Rollback Mechanism

Оптимистичное обновление интерфейса гарантирует отзывчивый UX у спасателей.
1. **Insert Pending**: При отправке сообщения в чат, оно мгновенно записывается в `ChatMessagesTable` со статусом `SyncStatus.pending`. Экран чата перерисовывается за 16мс.
2. **Server Confirmation**:
   - **Успех**: Сервер возвращает подтверждение по сокету или HTTP. Чат-сервис обновляет поле сообщения: `id` заменяется на серверный UUID, статус меняется на `SyncStatus.synced`.
   - **Сбой**: Сетевой клиент возвращает ошибку (например, блокировка пользователя или критический сбой API). Локальная запись удаляется (Rollback) либо её статус переходит в `SyncStatus.failed`, сигнализируя волонтеру восклицательным знаком о невозможности доставки сообщения.

## 5. Conflict Resolution & Consistency Matrix

| Сценарий конфликта | Merge Policy (Правило слияния) | Алгоритм разрешения |
|---|---|---|
| **Координаты трека** | **Append Logs** | Конфликт невозможен. Координаты пишутся как непрерывный кумулятивный хронологический лог. |
| **Параметры задач (Tasks)** | **Last-Write-Wins (LWW)** | Сверяются даты последнего изменения `updated_at`. Серверное значение приоритетно (Бэкенд — главный судья). |
| **Гео-границы (Polygons)** | **Server Authoritative** | Локальные изменения полиметров отсутствуют (волонтеры только читают полигоны координатора). База перезаписывает полигоны по свежей версии. |

## 6. WebSocket Synchronization Integration

WebSocket ускоряет синхронизацию:
- При разрыве сокета исходящие действия ставятся в классическую `SyncQueueTable` в базе.
- После реконнекта сокета, `SyncEngine` сначала проливает стаки накопленной очереди через классический HTTP/REST с уникальными ключами идемпотентности, чтобы бэкенд отсек возможные дубли.
- Только после полного сброса оффлайн-очереди сокет начинает в штатном режиме принимать броадкаст-события об изменениях (`chat.message.created`, `task.updated`), исключая нарушение хронологического порядка (race conditions).

## 7. Connectivity Handling Strategy (Умный сетевой сенсор)

Приложение отслеживает сетевой контур через нативный поток событий:
- При переходе сети во временное оффлайн-состояние, сетевые репозитории молча блокируют попытки обращения во внешнюю сеть, переключая вызовы только на чтение из локальной реляционной Drift базы.
- Переход спасателя в зону стабильной сотовой вышки мгновенно размыкает вентиль оффлайн-режима, пробуждает `SyncEngine` и форсирует пакетный сброс буферных таблиц.

## 8. Cache Invalidation & Data TTL Strategy

Во избежание раздутия локальной базы SQLite на устройстве волонтера действуют правила очистки Кэша:
- **Searches (Архивные Поиски)**: Завершённые поисковые операции, созданные более 7 дней назад, автоматически переносятся в архивный локальный статус или удаляются с устройства при периодической оптимизации базы данных на запуске.
- **Chat History**: Локально хранятся последние 300 сообщений поискового чата. Более старые сообщения архивируются и автоматически очищаются. Докачка истории происходит лениво (Lazy Loading Pagination) при прокрутке чата вверх.
- **Location Queue**: Координаты, помеченные как отправленные (`synced = 1`), удаляются из таблицы `LocationQueueTable` раз в 24 часа для минимизации файлового пространства телефона.
